# 理解度クイズ設計ガイドライン

長時間セッションで想像以上の量の実装が行われると、diff や計画を読むだけでは
挙動・設計意図を軽くしか把握できない。これを補うため、**実装者（人間）の理解度を測るクイズ**を
ゲートとして挟む。本ガイドラインは、そのクイズ設問を作るときの共通指針。

計画時と実装後で「問える材料」が違うため、**一つのクイズを使い回さず、役割の異なる二段のゲート**として持つ。

| | 実装前クイズ | 実装後クイズ |
|---|---|---|
| 組み込み先 | Plan スキル（spec-driven-dev） | 実装スキル（spec-implement） |
| 生成タイミング | plan 承認直前 | 実装完了後 |
| 材料 | requirements + plan + hearing-notes | diff + implementation-notes + hearing-notes |
| 問う対象 | 設計の意図・理由 | 実際にやった挙動・Deviations |
| ゲート位置 | plan → impl | push 前 |
| 落ちたら | 設計を固め直す | merge / push しない |

## enforcement の切り分け（重要）

- クイズの合否は基本 **自己申告**。hook（`exit 2`）で機械的に止めるのは飾りになりやすい。
- 機械判定できるもの（型 / lint）は型・lint に任せる。**理解ゲートは advisory に留める。**
- HTML テンプレートは 4択・YES/NO・並べ替えのいずれもスクリプトで正誤判定でき、
  **回答した時点で1問ずつ正誤と解説を表示する（即時フィードバック方式）**。
  これは advisory を保ったまま「どこを理解できていないかをその場で自覚させる」ためのもの。
  機械的ブロック（push 前 hook との連動）は導入していない。

## 即時フィードバックと「理解度測定」の両立

回答直後に正誤・解説を見せると学習効果は高い一方、素朴に実装すると
「正解が出るまで選び直せる」「前の解説が次の答えを教える」など、測定としては壊れやすい。
テンプレートは次のルールで両立させている。設問を作る側もこの前提を崩さないこと:

1. **初回回答でロック** — 一度回答した設問は答えを変更できない。スコアは常に
   「初回の回答」を反映するため、正解が出るまで選び直す抜け道がない。
2. **解説は回答後にのみ表示** — 回答前にヒントが見えることはない。
3. **並べ替えの初期配置はランダム** — 固定パターン（逆順など）だと並びを機械的に
   復元できてしまうため、毎回シャッフルする（偶然正解順で表示されることもない）。
4. **合否は初回の通し回答で判断** — 「リセット」は解説を読んだ後の再学習用。
   リセット後のスコアで合否を判断しない（この原則はテンプレートの advisory 文言にも明記している）。
5. **設問間で答えを漏らさない**（設問を作る側の責務） — 即時フィードバックでは、回答済み設問の
   正解・解説が画面に残ったまま次の設問に答える。ある設問の解説や選択肢が別の設問の答えを
   含まない構成にする。論点が依存し合うなら出題順を「解説が後続のヒントにならない」向きに並べるか、
   同一設問に統合する。

## 対話モード（quiz-output: interactive）の実施方法

`.config.yml` の `quiz-output` が `interactive` の場合、HTML を生成せず、セッション内で
**AskUserQuestion を使って1問ずつ出題**する。設問の作り方（出題観点・良い設問の条件・
セルフレビュー）は HTML 出力と共通。実施ルール:

- **1問ずつ出題し、回答直後に正誤と解説を伝えてから次の設問へ進む**（即時フィードバック）。
  解説では材料の該当箇所（どのファイルのどの記述が根拠か）に触れる。
- 選択肢の並びはシャッフルし、正解の位置を偏らせない。
- **初回の回答で正誤を確定する**。回答後に答えを変えさせない・同じ設問を出し直さない。
- 並べ替え（order）設問は AskUserQuestion では表現しづらいため、
  「正しい順序はどれか」を選ばせる choice（順序の候補を選択肢にする）に変換するか、対話モードでは使わない。
- 全問終了後、スコア（正解数 / 問題数）と passLine による合否、間違えた設問ごとに
  読み直すべき材料の該当箇所を提示する。
- 途中で不合格が確定しても最後まで出題する（どこを理解できていないかを全て炙り出すのが目的）。
- 対話できない実行環境（非対話実行）では `file` にフォールバックする。

## 設問セルフレビュー（生成後に必ず行う）

生成されたクイズが「微妙」になる典型パターンを踏まえ、出題（HTML 出力 / 対話開始）の前に
設問を1問ずつ自己レビューし、1つでも該当する設問は**修正するか捨てる**
（数合わせで残さない。捨てた結果 5 問を下回るなら材料に戻って作り直す）:

- **根拠を指させない** — 「この設問の正解の根拠は {ファイル名} の {該当記述}」と具体的に言えない
- **材料を読まなくても解ける** — 一般常識・消去法・選択肢の形（1つだけ長い / 具体的 / 丁寧）で正解できてしまう
- **暗記クイズになっている** — ファイル名・変数名・数値の暗記を問うだけで、間違えても理解不足を意味しない
- **「どう実装したか」偏重** — 手段・手順の再現を問う設問ばかりで、「その結果どう振る舞うか」
  「なぜその手段か」を問えていない（クイズ全体を見渡して、挙動・理由を問う設問が過半になっているか確認する）
- **識別子前提の設問文** — 設問や選択肢が関数名・変数名を知らないと読めない書き方になっている
  （挙動・役割の言葉に書き直し、識別子は括弧書きの補助に格下げする）
- **distractor が埋め草** — 誤答が明らかに的外れで、実質2択以下に退化している
- **どの機能にも使い回せる** — 「この機能の目的は？」のような、材料を差し替えても成立してしまう汎用設問
- **解説が正解の繰り返し** — なぜ正解なのか・誤答がなぜ違うのかを説明していない

推奨フロー: 出題観点ごとに多め（10問前後）にドラフトし、このチェックで削って 5〜10 問に絞る。

## 良い設問の条件

1. **材料に答えの根拠がある** — requirements / plan / hearing-notes / diff / notes に
   照らして一意に正解が決まる設問にする。憶測でしか答えられない設問は作らない。
2. **理解を測る、暗記を測らない** — 「変数名は何か」ではなく「なぜこの設計にしたか」
   「この制約が壊れると何が起きるか」を問う。
3. **もっともらしい誤答（distractor）** — 誤答は「ざっと読むと選びそうな別解釈・別設計」にする。
   明らかに的外れな選択肢は埋め草にしない。
4. **1設問1論点** — 複数の論点を1問に詰め込まない。
5. **落ちたら意味がある** — その設問を間違える＝「設計が固まっていない箇所」または
   「変更を理解できていない箇所」を指す、と言える設問にする。
6. **設問間の独立性** — 前の設問の正解・解説を読んだことが、後の設問の答えにならないようにする
   （即時フィードバック方式の前提。詳細は上の「即時フィードバックと『理解度測定』の両立」参照）。

## 設問タイプの使い分け

- **choice（4択）**: 設計判断の理由、データモデル選択の根拠、挙動の予測。最も汎用。
- **boolean（YES/NO）**: 「この制約が満たされないと壊れるか」「計画から逸脱したか」など、
  真偽がはっきりする命題。
- **order（並べ替え）**: データフロー・処理順序・状態遷移の理解。
  `items` は**正しい順序**で列挙する（テンプレートが表示時に自動シャッフルする）。

## 実装前クイズ（設計意図）の出題観点

- なぜこの設計にしたか（設計判断の理由）
- このデータモデル / 型インターフェースにした理由
- この制約が満たされないと何が壊れるか
- 変更されやすい箇所（data model / 型 / UX フロー）の意図
- 狙い: 自分の意図の確認と unknowns の炙り出し

各セクションの `brief` に設計判断の背景を先に解説し、その上で問う（解説 → 設問）。

## 実装後クイズ（実際の挙動）の出題観点

- 実装で実際に何が行われたか
- 途中で入った Deviations とその理由（implementation-notes 由来。**最低1問は必須**）
- 既存コードパスへの依存で発生した挙動（**diff の表面に出ない挙動を優先**）
- 想定 vs 実際: hearing-notes（事前の意図）と実装結果のズレ
- 狙い: 実際の作業結果の理解確認

**「どう実装したか」ではなく「その結果どう動くか」を問う。**
手段・手順の再現（「どの関数を追加したか」「どういう順で書き換えたか」）は diff を眺めれば
答えられる一方、正解しても挙動を理解している証拠にならない。問うべきは
「この入力・この状態のとき何が起きるか」「この変更で振る舞いはどう変わったか」
「なぜその手段を選んだか」。手段そのものを問うのは、選択が挙動・制約に影響する場合だけにする。

**設問文・選択肢は挙動・役割の言葉で書く。**
「`handleFooBar()` は何をするか」のような識別子前提の設問は、関数名を覚えていないと
答えられない暗記クイズになる。「保存ボタン押下時のバリデーションは〜」のように
挙動・役割で書き、関数名・変数名は必要なときに括弧書きの補助情報として添えるに留める。

hearing-notes（事前の意図）と implementation-notes（実際にやったこと）が別ファイルで
分かれているため、両方を読ませると「想定した設計 vs 実際の実装」のズレをそのまま出題材料にできる。

## 分量の目安

- セクション 2〜4、設問は合計 5〜10 問程度。多すぎると形骸化する。
- 設問タイプは choice に偏らせず、boolean / order も混ぜる。
- 合格ライン（`passLine`）は既定 0.8。機能の複雑さに応じて調整可。
