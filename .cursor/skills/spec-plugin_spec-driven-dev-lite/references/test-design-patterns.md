# テストデザインパターン

テスト戦略決定のための判断フレームワーク。
spec-planner が hearing-notes.md と exploration-report.md の情報から機能タイプを分類し、適切なテストパターンを自動選択する。

## 1. 機能タイプ分類

hearing-notes.md の目的・技術詳細と exploration-report.md の関連コード分析から、機能の主要タイプを判別する。
1つの機能が複数タイプに該当する場合は、すべての該当パターンを適用する。

### 分類チェックリスト

| タイプ | 判別条件 | 例 |
|--------|---------|-----|
| Pure Logic | 計算・変換・バリデーション・アルゴリズム | 価格計算、日付変換、入力検証 |
| API Integration | 外部API呼び出し・HTTPリクエスト・WebSocket | REST API連携、GraphQLクエリ |
| Data Transformation | データ形式変換・マッピング・集約 | CSV→JSON変換、DTOマッピング |
| State Management | 状態の作成・更新・削除・購読 | Redux store、Zustand store、DB操作 |
| UI Component | ユーザー操作・表示・フォーム | ボタン、モーダル、フォーム |
| Async Operations | Promise・タイマー・イベント・キュー | 非同期バッチ処理、ポーリング |
| Security/Auth | 認証・認可・トークン・暗号化 | JWT認証、RBAC、入力サニタイズ |
| Configuration | 設定変更・環境切り替え・フラグ | 環境変数、feature flag |
| DOM Manipulation | DOM読み書き・MutationObserver・CSS注入 | ブラウザ拡張、DOM操作ライブラリ |

## 2. タイプ別テストパターン決定マトリクス

| 機能タイプ | Unit Test | Integration Test | E2E Test | テスト優先度 |
|-----------|-----------|-----------------|----------|------------|
| Pure Logic | **必須** | - | - | 高 |
| API Integration | **必須**（モック） | **必須** | 推奨 | 高 |
| Data Transformation | **必須** | - | - | 高 |
| State Management | **必須** | 推奨 | - | 高 |
| UI Component | 推奨 | - | 推奨 | 中 |
| Async Operations | **必須** | 推奨 | - | 高 |
| Security/Auth | **必須** | **必須** | **必須** | 最高 |
| Configuration | **必須** | 推奨 | - | 中 |
| DOM Manipulation | 推奨 | - | 推奨 | 中 |

**凡例**: 必須 = 原則として省略不可 / 推奨 = 可能なら追加 / `-` = 通常不要

## 3. タイプ別テストシナリオ列挙

各タイプで列挙すべきテストシナリオ。テストTODOリスト作成時にこのリストから該当するものを選択する。
implementation-plan のテストTODOリストでは、テストファイルごとにセクションを作成し、各セクションにファイルの **役割** とテーブル形式（カテゴリ | テストケース | ユースケース/想定シナリオ | 期待結果）のTODOリストを記載すること。

### Pure Logic

- 正常系: 典型的な入力で期待結果を返す
- 正常系: バリエーション（複数の入力パターン）
- 境界値: 最小値・最大値・ゼロ・空
- 異常系: 無効な型・null/undefined
- エッジケース: 空配列・空文字列・特殊文字
- べき等性: 同じ入力で常に同じ出力（該当する場合）

### API Integration

- 正常系: 成功レスポンス（200系）の処理
- 異常系: クライアントエラー（4xx）の処理
- 異常系: サーバーエラー（5xx）の処理
- 異常系: ネットワークエラー・タイムアウト
- エッジケース: 空レスポンス・不正なJSON
- リトライ: リトライロジックの動作（該当する場合）
- 認証: トークン付与・リフレッシュ（該当する場合）
- **モック戦略**: 外部依存のモック/スタブ方針を明記

### Data Transformation

- 正常系: 標準的なデータ変換
- 正常系: 複数パターンの入力形式
- 境界値: 空データ・単一要素・大量データ
- 異常系: 不正な入力形式・欠損フィールド
- べき等性: 同じ入力で同じ出力
- パフォーマンス: 大量データでの処理時間（該当する場合）

### State Management

- 初期状態: デフォルト値の検証
- 状態遷移: 各アクション/イベントによる状態変化
- 副作用: 状態変更に伴う副作用の実行
- 競合: 同時更新・楽観的更新の整合性
- 永続化: 保存・復元の正確性（該当する場合）
- 購読: 状態変更の通知・再描画トリガー（該当する場合）

### UI Component

- レンダリング: 各状態（初期・ローディング・エラー・空・データあり）での表示
- インタラクション: クリック・入力・フォーカス・ホバー
- フォームバリデーション: 必須チェック・形式チェック・エラー表示
- アクセシビリティ: キーボード操作・aria属性・スクリーンリーダー
- レスポンシブ: 画面サイズ変化への対応（該当する場合）

### Async Operations

- 正常完了: 非同期処理の成功パス
- エラー: 非同期処理の失敗ハンドリング
- タイムアウト: 制限時間超過の処理
- キャンセル: 処理の中断・AbortController（該当する場合）
- 競合: 複数の非同期処理の実行順序・race condition
- リトライ: 再試行ロジック（該当する場合）

### Security/Auth

- 正常系: 正しい認証情報でアクセス成功
- 異常系: 無効なトークン・期限切れトークン
- 認可: 権限不足でのアクセス拒否
- 認可: ロール別アクセス制御（該当する場合）
- インジェクション: 不正な入力の無害化（XSS, SQL Injection等）
- セッション: セッションの作成・維持・破棄
- CSRF: クロスサイトリクエストフォージェリ対策（該当する場合）

### Configuration

- デフォルト値: 設定未指定時のフォールバック動作
- 値の切り替え: 設定変更の即時反映
- バリデーション: 不正な設定値の拒否・エラーメッセージ
- 環境差異: 環境ごと（dev/staging/prod）の設定切り替え
- 優先順位: 設定ソースの優先順位（環境変数 > ファイル > デフォルト等）

### DOM Manipulation

- 要素の存在: 対象DOM要素の検出・未検出
- 変更の反映: DOM操作後の状態確認
- 動的変化: MutationObserver等での動的要素検出
- クリーンアップ: イベントリスナー・Observer の解除
- 複数要素: 複数マッチ時の処理

## 4. テスト戦略の決定フロー

```
hearing-notes.md + exploration-report.md
    │
    ▼
機能タイプの分類（複数該当可）
    │
    ▼
┌─────────────────────────────────────────────────┐
│ Security/Auth を含む？                            │
│   → YES: TDD必須 + 全テストカテゴリ必須          │
│   → NO: 次へ                                     │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│ Pure Logic / Data Transformation /               │
│ State Management を含む？                        │
│   → YES: TDD推奨（Red-Green-Refactor）           │
│   → NO: 次へ                                     │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│ API Integration / Async Operations を含む？       │
│   → YES: 実装後テスト追加（モック戦略含む）       │
│   → NO: 次へ                                     │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│ UI Component / DOM Manipulation /                │
│ Configuration のみ？                             │
│   → 既存テストパターンがある: テスト追加          │
│   → 既存テストパターンがない: 手動検証のみ        │
└─────────────────────────────────────────────────┘
```

### テスト戦略の3パターン

| パターン | 適用条件 | tasks.md の構成 |
|---------|---------|----------------|
| **TDD** | Pure Logic, Data Transformation, State Management, Security/Auth | Red-Green-Refactor サイクルで構成 |
| **テスト追加** | API Integration, Async Operations, UI Component（テストインフラあり） | Implementation → Test → Verification |
| **手動検証のみ** | 純粋なUI/スタイリング変更、ロジック・外部依存なし | Verification に手動確認項目のみ |

### ユーザー指定によるオーバーライド

hearing-notes.md でユーザーが明示的にテスト方針を指定した場合は、上記フローの結論よりユーザー指定を優先する：
- 「TDDで進めたい」→ TDD を採用
- 「手動確認のみ」→ 手動検証のみ
- 「AIに任せる」→ 上記フローに従う

## 5. テスト不要の判断基準

以下の**すべて**に該当する場合のみ「自動テスト不要」と判断できる：

- 純粋な見た目の変更のみ（CSS・スタイリング・レイアウト）
- ロジック・状態管理を一切含まない
- 外部依存（API・DB・ファイルシステム等）がない
- 既存のテストで十分にカバーされている

上記に該当しても、**手動検証項目は必ず記載すること**。

## 6. 既存テストインフラへの適応

exploration-report.md の「テストインフラストラクチャ」セクションの情報を参照し、以下を決定する：

- テストファイルの命名規則（`*.test.ts` / `*.spec.ts` / `test_*.py` 等）
- テストファイルの配置場所（colocated / `__tests__/` / `tests/` 等）
- テストランナー（Jest / Vitest / pytest / cargo test 等）
- アサーションライブラリ（expect / assert / chai 等）
- モックライブラリ（jest.mock / vi.mock / sinon / nock 等）
- 既存テストの記述スタイルに合わせたテストコード
- カバレッジ設定がある場合はカバレッジ基準も考慮

**テストインフラ情報がない場合**: プロジェクトの技術スタックから標準的な選択肢を推奨する。
