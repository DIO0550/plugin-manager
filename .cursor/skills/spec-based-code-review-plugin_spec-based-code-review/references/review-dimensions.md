## 次元 1: 仕様充足性

**参照文書**: implementation-plan.md, tasks.md

### チェック項目

- [ ] `[NEW]` ファイルがすべて作成されているか
- [ ] `[MODIFY]` 変更がすべて適用されているか
- [ ] `[DELETE]` ファイルがすべて削除されているか
- [ ] tasks.md の完了状態（■）と実際のファイル状態が一致しているか
- [ ] 未完了タスク（□）が残っている場合、その理由が明確か

### 典型的な指摘

- CRITICAL: 「implementation-plan に記載された X が未実装」
- WARNING: 「tasks.md では完了(■)だが、実装が不完全」

### 典型的な指摘

- CRITICAL: 「hearing-notes の目的 X に対応する実装が見当たらない」

## 次元 3: 設計方針遵守

**参照文書**: ユニバーサルな設計原則（primary）, implementation-plan.md / exploration-report.md（spec記載の補足用, optional）

### チェック項目

**ユニバーサル（spec不要）**:
- [ ] **SRP**: 各モジュール/関数が単一の責務を持ち、変更理由が1つだけか
- [ ] **DIP**: 上位モジュールが下位モジュールの具体実装に直接依存していないか（抽象に依存しているか）

**spec拡張（spec文書がある場合に追加）**:
- [ ] データフロー図と実装のデータの流れが一致しているか
- [ ] 状態遷移図と状態管理の実装が一致しているか
- [ ] API設計（エンドポイント・型・レスポンス）と実装が一致しているか
- [ ] コンポーネント分割方針に従っているか
- [ ] 命名規則が既存パターンと一致しているか

**注意**: spec文書で意図的に SRP/DIP に反する設計が選択されている場合は、指摘に `spec記載` として補足する。

### 典型的な指摘

- CRITICAL: 「implementation-plan のデータフローでは A→B→C だが、実装では A→C に直結」
- WARNING: 「exploration-report の命名規則（camelCase）に対して snake_case を使用」
- WARNING: 「モジュール X が3つの異なる責務を持ち、変更理由が複数存在する（SRP違反）」
- WARNING: 「上位モジュールが下位モジュールの具体クラスを直接 import している（DIP違反）」
- INFO: 「SRP に反するが、implementation-plan で意図的にこの構成が選択されている」（spec記載として補足）

## 次元 5: 意図的複雑性の保護（全エージェント共通）

**参照文書**: 全計画ドキュメント（参考）

### 判定フロー

```
コードが冗長/複雑に見える
  ↓
spec文書があるか？
  ↓
[specあり] specの前提を確認
  specに前提がある → その前提は技術的に妥当か？
    妥当 → INFO（spec記載として補足: 前提が妥当なため現状維持推奨）
    疑わしい → WARNING（specの前提自体への懸念を併記）
  specに前提がない → WARNING（簡素化を提案）
  ↓
[specなし] ユニバーサル原則で判断
  明確なアンチパターン（KISS違反等） → WARNING
  判断がつかない → INFO（改善の余地として提示、断定は避ける）
```

### specの前提として確認するパターン

| 表面的な印象 | specの前提（妥当性を確認の上でspec記載として補足） |
|-------------|-------------|
| エラーハンドリングが冗長 | hearing-notes でエッジケース網羅が要求 |
| ループが分離されている | implementation-plan で可読性を優先 |
| テストケースが多い | テストTODOリストに個別シナリオが列挙 |
| 型定義が冗長 | exploration-report の既存パターンに合致 |
| 分岐が多い | hearing-notes のエッジケース一覧と対応 |
| 似たコードが繰り返されている | implementation-plan で意図的に個別処理 |
| 似たバリデーションが複数箇所にある | 異なるドメイン概念（ユーザー vs 商品等）を扱い、独立に変化する |
| 同じ計算ロジックが重複している | 異なるコンテキスト（注文 vs 見積等）で将来分岐する可能性がある |

### WARNING の典型パターン

| 表面的な印象 | 判断理由 |
|-------------|---------|
| 未使用のコードが残っている | どの要件にも対応しないデッドコード |
| 完全に同一のロジックが重複 | 同一ドメイン概念に起因し、分離する理由がない |
| コメントアウトされたコード | 不要な実験的コード |
| specで選択された設計だが前提が疑わしい | specの判断自体に技術的な問題がある（spec自体への指摘として併記） |

### 誤DRY検出フロー

AIレビューアーは表面的なコード類似を見て「共通化すべき」と提案しやすい。この傾向を構造的に抑制するため、以下の3ステップで判定する。

```
表面的に類似したコードが存在する
  ↓
Step 1: 変更理由テスト（ユニバーサル — spec不要）
  「この2つのコードは、同じビジネス上の理由で同時に変更されるか？」
  NO/不明 → INFO（個別実装を維持。specありの場合はspec記載として補足）
  YES → Step 2 へ
  ↓
Step 2: ドメイン概念テスト（ユニバーサル — spec不要）
  「この2つのコードは、同一のドメイン概念（同じビジネスルール）を表現しているか？」
  NO → INFO（個別実装を維持。specありの場合はspec記載として補足）
  YES → Step 3 へ
  ↓
Step 3: 仕様確認テスト（spec文書がある場合のみ）
  「implementation-plan で意図的な個別実装が指定されているか？」
  YES → INFO（仕様が個別実装を正当化。spec記載として補足）
  NO → INFO（共通化の検討を提案 — WARNING/CRITICAL にはしない）
  spec文書がない場合 → INFO（共通化の検討を提案）
```

#### 概念的同一性の判定基準

| 判定項目 | 同一と判定 | 異なると判定 |
|---------|----------|------------|
| 変更理由 | 同じ理由で同時に変わる | 異なる理由で別々に変わる |
| ドメイン概念 | 同じビジネスルールを表現 | 異なるビジネスルールが偶然同じ形 |
| 将来の分岐可能性 | 今後も同じ振る舞いを維持する蓋然性が高い | 独立に進化する可能性がある |
| 仕様上の位置づけ | 同一の仕様要件に起因 | 異なる仕様要件に起因 |

## 次元 9: アーキテクチャ整合性

**参照文書**: ユニバーサルな設計原則（primary）, implementation-plan.md / exploration-report.md（spec記載の補足用, optional）

### チェック項目

**ユニバーサル（spec不要）**:
- [ ] 依存関係の方向が正しいか（循環依存の検出）
- [ ] **KISS**: 要件を満たすために必要十分な複雑さか。不必要な抽象化レイヤーが導入されていないか
- [ ] **KISS**: 抽象化レイヤーの各層に明確な存在理由があるか（現在1つの実装しか持たないパターンは過剰でないか）
- [ ] **ファイル凝集度**: 1ファイル1関数の過剰分割をしていないか。関数名（`getUserName.ts`）ではなく概念（`user.ts`）でファイルを構成しているか。すり抜けバリアントとして `{関数名.ts}/index.ts`（ディレクトリ名に拡張子）も検出すること
- [ ] **ファイル凝集度**: 関連する処理（同一ドメイン概念に属する関数群）が同一ファイルに凝集しているか

**spec拡張（spec文書がある場合に追加）**:
- [ ] 既存の設計パターンに従っているか
- [ ] 状態管理の方針と一致しているか
- [ ] API設計の規約に従っているか

**KISS と意図的複雑性の保護の関係**: KISS に基づいて「シンプルにすべき」と判断する前に、spec文書がある場合は次元5の判定を行うこと。implementation-plan で将来の拡張に備えて設けられた構造はspec記載として補足し、KISS の名の下に安易に削除提案してはならない。

### 典型的な指摘

- CRITICAL: 「循環依存が発生している」
- WARNING: 「既存パターンと異なる状態管理アプローチ」
- WARNING: 「要件を満たすのに不要な抽象化レイヤーが導入されている（implementation-plan に根拠なし）」
- WARNING: 「1ファイル1関数で過剰分割されている — `formatUserName.ts`, `validateUserEmail.ts` は概念的に `user.ts` に凝集すべき」
- WARNING: 「関数名でファイルを命名している（`calculateTax.ts`）— 概念名で命名すべき（`pricing.ts` 等）」。すり抜けバリアント: `calculateTax.ts/index.ts` のようにディレクトリ化して同じ問題を持ち込むケースも同様に WARNING
- INFO: 「コンポーネント粒度が既存より細かいが、implementation-plan の方針に合致」（spec記載として補足）
- INFO: 「一見過剰な抽象化だが、implementation-plan で将来の拡張に備えて設けられた構造」（spec記載として補足）

### 普遍的コード品質ルール

以下のアンチパターンはspec文書の有無に関わらず常に指摘する（仕様が正当化する理由が原理的に存在し得ないため）。詳細は `finding-classification.md` を参照。

| パターン | 分類 | 説明 |
|---------|------|------|
| 循環依存 | CRITICAL | モジュール間の循環参照。常にアーキテクチャ上の問題 |
| 拡張子付きディレクトリ名 | WARNING | 「関数名によるファイル命名」のすり抜けバリアント。`{関数名.ts}/index.ts` のようにディレクトリ名に拡張子が含まれている。ファイル名チェックを回避する形で同じ問題を持ち込む |

## 次元 10: モック制限（MOCK-SCOPE）

**参照文書**: implementation-plan.md（テスト戦略セクション）, exploration-report.md（テストインフラセクション）

**詳細ルール**: `references/test-review-rules.md` を参照

### 原則

モックはテストと実装の間に結合を生む。内部モジュールをモックすると、振る舞いが保持されていてもリファクタリングでテストが壊れる。モックはテスト対象と外部世界の境界でのみ使用する。

### チェック項目

- [ ] `vi.mock(...)` / `jest.mock(...)` の対象が外部依存のみか
- [ ] `vi.spyOn(...)` / `jest.spyOn(...)` で内部モジュールをモック化していないか
- [ ] `vi.fn()` / `jest.fn()` が内部モジュールの代替として使われていないか
- [ ] implementation-plan のモック方針と実際のモック使用が一致しているか

### 典型的な指摘

- CRITICAL: 「内部モジュール `./services/UserService` をモックしている — 実モジュールを使ってテストすべき」
- WARNING: 「`./repositories/UserRepository` をモックしているが、これがアーキテクチャ上の外部境界なら許容」
- INFO: 「内部モジュールをモックしているが、implementation-plan のテスト戦略で『Repository 層はモックして単体テストする』と明記されている」（spec記載として補足）

## 次元 11: 振る舞いテスト（BEHAVIOR-TEST）

**参照文書**: implementation-plan.md（テスト戦略セクション）, hearing-notes.md

**詳細ルール**: `references/test-review-rules.md` を参照

### 原則

テストはシステムが「何をするか」（観察可能な振る舞い）を検証すべきであり、「どうやるか」（実装詳細）を検証すべきではない。これによりテストがリファクタリング耐性を持つ。

### チェック項目

- [ ] 内部コラボレーターへの spy アサーション（`toHaveBeenCalled*`）がないか
- [ ] プライベートメソッド・フィールドへの直接アクセスがないか
- [ ] テストが戻り値・状態変化・例外など観察可能な振る舞いを検証しているか
- [ ] 外部境界への spy アサーションは CRITICAL にしない（観察可能な振る舞い）

### 典型的な指摘

- CRITICAL: 「内部メソッド `calculator.internalCalculate` への spy アサーション — 戻り値で振る舞いを検証すべき」
- CRITICAL: 「プライベートフィールド `obj['_privateField']` を直接テストしている — 公開APIを通じて検証すべき」
- WARNING: 「`result._metadata.processingSteps` は出力契約の一部かどうか要判断」
- INFO: 「内部メソッドの呼び出しを検証しているが、implementation-plan で『この内部メソッドの呼び出しは振る舞いの一部として保証する』と明記されている」（spec記載として補足）

## 次元 12: テスト価値（TEST-VALUE）

**参照文書**: implementation-plan.md（テスト戦略セクション）

**詳細ルール**: `references/test-review-rules.md` を参照

### 原則

テストはリグレッション保護を提供するために書く。ロジックのないコード（分岐・ループ・計算・条件判定がないコード）をテストしても、壊れる可能性がないためリグレッション保護の価値がない。そのようなテストは保守コストだけが発生する。

### チェック項目

- [ ] テスト対象のコードにロジック（分岐・ループ・計算・条件判定・例外送出）があるか
- [ ] ロジックのないコードに対してテストが書かれていないか
- [ ] テストが既存の別テストと実質的に同じことを検証していないか

### 典型的な指摘

- WARNING: 「ロジックのない単純コンストラクタ `new User(name, email)` のテスト — プロパティ代入のみでリグレッション保護の価値がない」
- WARNING: 「単純な getter `getFullName()` のテスト — `return this.firstName + ' ' + this.lastName` に分岐がなく壊れる可能性が極めて低い」
- WARNING: 「パススルーメソッド `delegate.execute(args)` のテスト — 内部で別メソッドを呼ぶだけでロジックがない」
- INFO: 「単純に見えるが、implementation-plan のテストTODOリストに明示的に含まれている」（spec記載として補足）

## 次元 13: テストケース網羅性（TEST-COVERAGE）

**参照文書**: implementation-plan.md（テスト戦略セクション・テストTODOリスト）, hearing-notes.md

**詳細ルール**: `references/test-review-rules.md` を参照

### 原則

ロジックのあるコード（分岐・ループ・計算・例外送出）にはテストが必要。特に条件分岐の各パスと境界値はリグレッションが発生しやすいポイントであり、テストケースが不足しているとバグの検出が遅れる。

### チェック項目

- [ ] 条件分岐の各パス（正常系・異常系・エッジケース）にテストがあるか
- [ ] 境界値（0, 1, 上限, 空配列, null/undefined 等）がテストされているか
- [ ] 例外送出の条件がテストされているか
- [ ] 状態遷移のある処理で、各遷移パスがテストされているか
- [ ] implementation-plan のテストTODOリストの項目が漏れなくテストされているか

### 典型的な指摘

- CRITICAL: 「`calculateDiscount(amount)` に3つの分岐（通常/会員/VIP）があるが、通常の1パスしかテストされていない」
- CRITICAL: 「implementation-plan のテストTODOリスト項目『タイムアウト時のリトライ』に対応するテストが存在しない」
- WARNING: 「`validateAge(age)` の境界値（0, 負数, 上限値）がテストされていない」
- WARNING: 「`OrderStatus` の状態遷移で `pending → cancelled` パスのテストがない」
- INFO: 「テストケースが少ないが、implementation-plan のテスト戦略で『正常系のみテスト、異常系は手動確認』と明記されている」（spec記載として補足）

## 次元 14: コメント品質 — why原則（COMMENT-WHY）

**参照文書**: ユニバーサルなコメント原則（primary）, implementation-plan.md / hearing-notes.md（spec記載の補足用, optional）

### 原則

コメントはコードから読み取れない「なぜ」（制約・理由・注意喚起）を説明すべき。コードをなぞるだけのコメント（what / how）は、コードの変更に追従されず乖離してノイズになる。変更履歴はバージョン管理システムが担うべき情報。

### チェック項目（全てユニバーサル — spec不要）

- [ ] コードが何をするかをなぞるだけの what コメントがないか
- [ ] 処理手順をコードと重複して説明する how コメントがないか
- [ ] 変更履歴・作業者向けコメント（「〜を修正」「レビュー対応」「日付 変更」等）がないか
- [ ] 外部依存コメント（Issue/PR番号・外部URL/チケットID・人名/会話への依存）がないか
- [ ] コメントアウトされたコードが残っていないか（次元5と共通）

**判定基準**: 「このコメントを削除したとき、コードだけでは読み取れない情報が失われるか？」
- YES → whyコメント（保護 — 指摘しない）
- NO → what / how コメント（WARNING）

**外部依存コメント**: 意図がコメント本文になく、外部リソース（Issue/PR・URL・チケット・人/会話）を見に行かないと「なぜ」が分からないコメントは WARNING。`// #123 対応`・`// JIRA-1234`・`// 〇〇さんの指摘で` 等が該当。ただし本文に「何を・なぜ」が書かれ、参照が補足に留まる場合は保護（`// バグ回避のため逐次実行（#123）` は適切）。

**whyコメントの保護**: 制約の説明・設計判断の理由・注意喚起・却下した代替案や危険な単純化の説明（Why-not: 「素朴な Map に置き換えるとメモリリークが再発する（#482）」等）・順序依存やセキュリティ境界など壊し方が非自明な箇所の局所的な理由・一見不自然なコードの正当化、および意図本文があり参照が補足に留まるものは適切なコメントであり、指摘しない。

### 典型的な指摘

- WARNING: 「コードをなぞるだけのwhatコメント（`// ユーザーを取得する` + `getUser()`）— コメントを削除してもコードだけで読み取れる」
- WARNING: 「変更履歴コメント（`// 2024-01-01 バグ修正`）— バージョン管理システムが担うべき情報」
- WARNING: 「外部依存コメント（`// #234 対応`）— Issue番号のみで意図が読めない。本文に『何を・なぜ』を記述すべき」
- INFO: 「冗長に見えるコメントだが、implementation-plan でエッジケースの根拠として明記されている」（spec記載として補足）

## 次元 15: docコメント整合性（DOC-CONSISTENCY）

**参照文書**: ユニバーサルなコメント原則（primary）, exploration-report.md（既存のdocコメント規約の参考用, optional）

### 原則

docコメント（JSDoc / TSDoc / docstring / Javadoc 等）は関数の公開契約を記述する。実装とズレたdocコメントは読者を誤導し、コメントがない状態より有害。整合性の検証はdiffだけでなく関数全体を読んで行うこと。

### チェック項目（全てユニバーサル — spec不要）

- [ ] `@param` の名前・数が実シグネチャと一致しているか
- [ ] `@returns` の記述（型・意味）と実際の戻り値が一致しているか
- [ ] `@throws` の記述と実際の例外送出が一致しているか
- [ ] docコメントの説明文が実装の振る舞いと矛盾していないか
- [ ] ロジック（分岐・ループ・計算・例外送出）のある公開関数にdocコメントがあるか（欠如は INFO）

### 典型的な指摘

- WARNING: 「docコメントの `@param userId` が実シグネチャの `id` と不一致 — リネームへの追従漏れ」
- WARNING: 「docコメントは『見つからない場合は null を返す』と記述しているが、実装は例外を投げる」
- INFO: 「3つの分岐を持つ公開関数 `calculateDiscount` にdocコメントがない」
- INFO: 「docコメントの形式が exploration-report の既存規約（TSDoc）と異なる」（spec記載として補足）

## 次元 16: 不要なフォールバック・防御的コード（DEFENSIVE-CODE）

**参照文書**: ユニバーサルな原則（primary）, hearing-notes.md / implementation-plan.md（フォールバックが仕様かの確認用, optional）

### 原則

仕様が要求していない「念のため」のフォールバックや防御的コードは書かない。不要な分岐はコードを濁らせ、テストされない死んだ経路を増やし、実際の失敗（DB接続断・データ不整合等）を握りつぶして呼び出し元で区別不能にする。エラー処理は仕様が要求する場所（システム境界・エントリポイント・ハンドラ）にだけ置き、それ以外では失敗をそのまま伝播させる。

フォールバックが本当に仕様（合意済みの劣化提供）なら、次の3点セットを必ず揃える:

1. なぜ許容するのかを書いたコメント（Why と出典）
2. フォールバック発生に気づける観測手段（ログ、メトリクス）
3. その挙動を固定するテスト

### チェック項目

- [ ] 起こりえないエラーへの try-catch がないか
- [ ] 呼び出し元が処理すべき例外をその場で catch して null/既定値に変換していないか（例外の握りつぶし）
- [ ] 型やバリデーションで保証済みの値への「とりあえず」の null チェック・既定値（`?? ""` 等）がないか
- [ ] 失敗時に黙って別実装・既定値へ切り替える「動かすための」フォールバックがないか
- [ ] 仕様としてのフォールバックに3点セット（コメント+出典 / 観測手段 / テスト）が揃っているか

### 判定フロー

```
フォールバック/防御的分岐を発見
  ↓
仕様（hearing-notes / implementation-plan）がこの劣化挙動を要求しているか？
  ↓
[要求している] 3点セット（コメント+出典 / 観測手段 / テスト）が揃っているか？
  揃っている → 指摘しない（またはINFOでspec記載として補足）
  不足している → WARNING（不足要素を明示）
  ↓
[要求していない / spec文書がない]
  例外の握りつぶし・型保証済み値への防御・黙示のフォールバック → WARNING（削除して伝播を提案）
  判断がつかない境界的なケース → INFO（断定は避ける）
```

**次元5との関係**: 「エラーハンドリングが冗長」に見えるコードは、まず次元5の判定（specの前提確認）を行うこと。hearing-notes がエッジケース網羅を要求している場合、それは防御的コードではなく仕様である。この次元が対象とするのは、どの仕様要件にも対応しない「念のため」の分岐のみ。

### 典型的な指摘

- WARNING: 「`findById` は未検出時に null を返す設計だが try-catch で握りつぶしている — DB接続断も『ユーザー不在』に変換され、呼び出し元で区別できない」
- WARNING: 「型定義上必須の `user.profile` に `?.name ?? ''` の防御 — データ不整合を空文字でごまかしており、失敗させるべき」
- WARNING: 「設定ファイル読込失敗時に黙って DEFAULT_CONFIG で続行 — 仕様にない劣化挙動で、観測手段もテストもない」
- WARNING: 「hearing-notes で合意されたフォールバックだが、観測手段（ログ・メトリクス）がなく発生に気づけない」
- INFO: 「一見不要な防御に見えるが、hearing-notes でエッジケース網羅が要求されている」（spec記載として補足）
