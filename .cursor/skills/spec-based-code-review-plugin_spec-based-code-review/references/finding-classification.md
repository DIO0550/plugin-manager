# 指摘分類ルール

コードレビューにおける指摘の分類基準。全エージェント共通で適用する。

---

## 分類一覧

| 分類 | 意味 | アクション | アイコン |
|------|------|-----------|---------|
| CRITICAL | 重大な問題 | 必ず修正 | 🔴 |
| WARNING | 要確認の潜在的問題 | 基本修正 | 🟡 |
| INFO | 参考情報 | 任意対応 | 🔵 |

---

## 根拠の種類

| 種類 | 説明 | spec文書 |
|------|------|---------|
| ルール根拠 | ユニバーサルな設計・パフォーマンス・テスト原則への違反（SRP, DIP, KISS, N+1等） | 不要 |
| 仕様根拠 | spec文書（implementation-plan, hearing-notes等）からの具体的引用 | 必要 |

ユニバーサルな原則はそれ自体が根拠となる。spec文書があれば参考にする。

### spec文書の位置づけ

spec文書はあくまで参考。spec自体が間違っている可能性もあるため、specに記載があるというだけで問題を見逃してはならない。specの設計判断に疑問がある場合は、spec自体への指摘（INFO または WARNING）を出す。

specに関連する記載がある場合、指摘の `spec記載` フィールドに添える。ただしspecの記載で指摘の分類は変えない。

---

## CRITICAL: 重大な問題

### 定義

ユニバーサルな原則への重大な違反、または仕様に反する実装がある。

### 該当条件

**ルール根拠（spec不要）**:
- 循環依存の発生
- N+1問題
- メモリリーク（イベントリスナー未解除、クロージャによる参照保持等）
- セキュリティ上の問題

**仕様根拠（spec文書がある場合）**:
- implementation-plan の `[NEW]` ファイルが作成されていない
- implementation-plan の `[MODIFY]` 変更が適用されていない
- DoD の項目が満たされていない
- hearing-notes のスコープ内の機能が未実装
- テストTODOリストのシナリオがテストされていない

### 出力フォーマット

```markdown
### C-{NNN}: {タイトル}

- **次元**: {担当次元名}
- **対象**: `{ファイルパス}` L{行番号}
- **根拠**: {ルール根拠: 違反しているユニバーサル原則 / 仕様根拠: spec文書名 Section X: "引用"}
- **問題**: {問題の説明}
- **spec記載**: {specに関連する記載があれば補足。なければ省略}
- **改善案**: {具体的な修正提案}
```

## WARNING: 要確認

### 定義

潜在的な問題があり確認が必要。ユニバーサル原則違反または仕様からの逸脱。

### 該当条件

**ルール根拠（spec不要）**:
- SRP/DIP 違反
- KISS 違反 — 不必要な抽象化・間接参照
- 1ファイル1関数の過剰分割、関数名によるファイル命名（凝集度の問題）。すり抜けバリアント: `{関数名.ts}/index.ts`（ディレクトリ名に拡張子）も同様
- パフォーマンス上の懸念（不要な再計算、並列化の機会等）
- whyを超えたコメント — コードをなぞるだけのwhatコメント、howの逐語説明、変更履歴コメント（VCSが担うべき情報）
- docコメントと実装の不整合 — `@param`/`@returns`/`@throws` や説明文がシグネチャ・振る舞いと一致しない
- 不要なフォールバック・防御的コード — 例外の握りつぶし（catch して null/既定値で続行）、型・バリデーションで保証済みの値への「とりあえず」の既定値、失敗時に黙って別実装へ切り替える仕様にないフォールバック、仕様フォールバックの3点セット不足（コメント+出典 / 観測手段 / テスト）

**仕様根拠（spec文書がある場合）**:
- hearing-notes のスコープ外の実装が含まれている
- exploration-report の既存パターンと異なる実装
- テストファイルの配置が規約と異なる
- DoD 項目が部分的にのみ満たされている

### 出力フォーマット

```markdown
### W-{NNN}: {タイトル}

- **次元**: {担当次元名}
- **対象**: `{ファイルパス}` L{行番号}
- **根拠**: {ルール根拠: 違反しているユニバーサル原則 / 仕様根拠: spec文書名 Section X: "引用"}
- **問題**: {問題の説明}
- **spec記載**: {specに関連する記載があれば補足。なければ省略}
- **改善案**: {具体的な修正提案}
```

## INFO: 参考情報

### 定義

問題ではないが、将来の保守やリファクタリングの参考になる観察事項。

### 該当条件

- 既存パターンとは異なるが改善として妥当な実装
- 将来的な拡張ポイントの示唆
- ドキュメントの追加が有益な箇所
- ロジック（分岐・ループ・計算・例外送出）のある公開関数にdocコメントがない

### 出力フォーマット

```markdown
### I-{NNN}: {タイトル}

- **対象**: `{ファイルパス}` L{行番号}
- **根拠**: {ルール根拠 or 仕様根拠}
- **観察**: {観察内容}
- **spec記載**: {specに関連する記載があれば補足。なければ省略}
```

## 指摘時の根拠確認ガイド

### ユニバーサル原則で判断可能（ルール根拠で指摘可）

以下はspec文書がなくてもルール根拠で指摘できる。spec文書があれば参考にし、関連する記載を `spec記載` フィールドに添える。

| 指摘 | ルール根拠 | specがある場合の補足 |
|------|----------|-------------------|
| 「SOLID原則に違反している」 | SRP/DIP原則 | 計画で意図的に選択された設計か確認 → spec記載として補足 |
| 「不必要に複雑」(KISS) | KISS原則 | 計画で将来の拡張に備えた構造か確認 → spec記載として補足 |
| 「このループは非効率です」 | パフォーマンス原則 | 計画で可読性優先の選択か確認 → spec記載として補足 |
| 「コンポーネントが大きすぎる」 | SRP原則 | 計画でこの粒度が決められた理由を確認 → spec記載として補足 |
| 「この抽象化は不要」 | KISS原則 | 計画でこの抽象レイヤーの目的を確認 → spec記載として補足 |
| 「コードが重複している」 | 誤DRY検出の判定フロー（変更理由テスト → ドメイン概念テスト）を通過した場合のみ INFO | specがあれば Step3（仕様確認テスト）も実施 → spec記載として補足 |
| 「この関数は独立ファイルにすべき」 | ファイル凝集度の原則（概念的な凝集度を損なう場合は指摘しない） | 計画のファイル分割方針を確認 → spec記載として補足 |

### spec文書がある場合のみ判断可能

以下はspec文書の記載と照合しないと判断できない。spec文書がない場合はこれらの指摘は出さない。

| 指摘 | 必要な仕様根拠 |
|------|--------------|
| 「このコードは冗長です」 | 仕様のどの要件が不要か？ |
| 「もっとシンプルにできます」 | シンプル化しても仕様を満たせるか？ |
| 「このテストは多すぎます」 | テストTODOリストのどの項目が不要か？ |
| 「この分岐は不要です」 | この分岐が処理するエッジケースは仕様で不要とされているか？ |
| 「別のライブラリを使うべき」 | exploration-report のライブラリ選定根拠は？ |
| 「メモ化すべきです」 | 再計算がパフォーマンス要件を満たさない根拠は？ |

**すべての指摘には「根拠」フィールドを必須で含める。** ルール根拠（ユニバーサル原則名）または仕様根拠（spec文書からの引用）のいずれかを記載する。

## 常に指摘可能なアンチパターン

以下のアンチパターンは、spec文書の有無に関わらず常に指摘する。

| パターン | 分類 | 根拠 |
|---------|------|------|
| 循環依存 | CRITICAL | モジュール間の循環参照。常にアーキテクチャ上の問題 |
| セキュリティ上の問題 | CRITICAL | XSS, SQL Injection, 認証バイパス等 |
| 拡張子付きディレクトリ名（`{関数名.ts}/index.ts`） | WARNING | 「関数名によるファイル命名」のすり抜けバリアント。ディレクトリ名に拡張子を付ける正当な理由は存在しない |

**追加基準**: このリストは最小限に保つこと。新しいパターンを追加する際は、以下をすべて満たすこと:
1. プロジェクト・言語を問わず**常に**問題である
2. 仕様が正当化する理由が**原理的に存在し得ない**
3. ツール/エディタの誤操作で発生する（意図的に選択されることがない）

---

## 指摘番号の採番ルール

各エージェントは自分の出力内で独立に番号を振る:

- `C-001`, `C-002`, ... (CRITICAL)
- `W-001`, `W-002`, ... (WARNING)
- `I-001`, `I-002`, ... (INFO)

オーケストレーターが統合レポートを生成する際に、全エージェントの指摘を通し番号で再採番する。
