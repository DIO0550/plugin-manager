# フィードバック明確化プロトコル

ユーザーが修正を要求した際、フィードバックの明確性を判定し、
曖昧な場合は AskUserQuestion で具体化してから修正に進む。

## 判定基準

フィードバックが以下のいずれかに該当する場合、「曖昧」と判定する。

### 1. 対象が不明確

「ここ」「この辺」「この部分」など指示語のみで、具体的なセクション・ファイル・タスクが特定できない。

- 例: 「ここを直して」「この辺がおかしい」「あの部分を変えて」

### 2. 方向性が不明確

「もっと良く」「もう少し」「ちゃんと」など程度・方向を示さない修飾語のみ。

- 例: 「もうちょっと良くして」「もう少し詳しく」「ちゃんとして」「微妙」

### 3. 前回決定との矛盾

hearing-notes に記録された決定事項や、exploration-report の制約と矛盾する要求。

- 例: ヒアリングで「ローカルストレージ」を選択したが「DBに保存して」と要求

### 4. 複数対象の優先度不明

複数箇所への修正を一度に要求し、優先順位や依存関係が不明。

- 例: 「AとBとCを全部変えて」「いろいろ直して」

## 明確なフィードバックの例（そのまま修正に進む）

- 「tasks.md のタスク3を削除して、代わりにXXXを追加して」
- 「implementation-plan の変更案セクションで、YYY関数のシグネチャをZZZに変更して」
- 「データフロー図にキャッシュレイヤーを追加して」
- 「テスト方針をTDDからインテグレーションテスト中心に変えて」

## AskUserQuestion パターン

### 対象不明確の場合

```
question: "どの部分を修正しますか？"
header: "修正対象"
options:
  - label: "設計方針"
    description: "implementation-plan のアーキテクチャ・設計判断を修正"
  - label: "変更案（コード）"
    description: "implementation-plan の具体的なコード変更案を修正"
  - label: "システム図"
    description: "状態マシン図・データフロー図を修正"
  - label: "タスク構成"
    description: "tasks.md のタスク分割・順序を修正"
```

### 方向性不明確の場合

```
question: "どのように修正しますか？"
header: "修正方向"
options:
  - label: "もっと詳細に記述する"
    description: "説明やコードスニペットを追加して具体化"
  - label: "より簡潔にまとめる"
    description: "冗長な部分を削除してポイントを絞る"
  - label: "別のアプローチに変更する"
    description: "設計方針や技術選定を見直す"
  - label: "具体例・コードスニペットを追加する"
    description: "実装イメージが湧くようサンプルコードを追加"
```

### 矛盾の場合

```
question: "{矛盾する2つの方針}のどちらを採用しますか？"
header: "方針確認"
options:
  - label: "{hearing-notesの元の決定} を維持"
    description: "ヒアリング時の決定を優先する"
  - label: "{新しい要求} に変更する"
    description: "今回の要求に合わせて計画を修正する"
  - label: "両方を検討して比較案を作成する"
    description: "トレードオフを整理してから決定する"
```

### 複数対象の場合

```
question: "どれを優先的に修正しますか？（残りは順次対応します）"
header: "優先順位"
options:
  - label: "{ユーザーが挙げた項目A}"
    description: "{項目Aの概要}"
  - label: "{ユーザーが挙げた項目B}"
    description: "{項目Bの概要}"
  - label: "{ユーザーが挙げた項目C}"
    description: "{項目Cの概要}"
  - label: "すべて同時に修正する"
    description: "全項目をまとめて修正（依存関係を考慮）"
```

## 運用ルール

1. **判定は1回のみ** — AskUserQuestion の回答後は即座に修正に進む（再質問ループしない）
2. **疑わしきは質問する** — 判定に迷った場合は「曖昧」として扱う
3. **Other回答の尊重** — ユーザーが Other で自由記述した場合はその内容に従う
4. **カテゴリの複合** — 複数カテゴリに該当する場合は、最も重要な1カテゴリのみ質問する（最大4問の制約を守る）
